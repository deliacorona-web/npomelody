## НПО Мелодия — фронтенд + админка

Статический фронтенд с Supabase как backend (DB/Auth/Storage) и отдельной админ-панелью. Сборка и локальный запуск через Vite, деплой возможен на GitHub Pages или любой статический хостинг.

### Структура

- `index.html` — публичный сайт (Афиша, Артисты, Релизы, Трансляции, Мерч, Профиль, Контакты)
- `admin.html` — админ-панель (дашборд, пользователи, события, артисты, др.)
- `src/`
  - `main.js` — enty для публичного сайта
  - `admin/main.js` — entry для админки
  - `db.js` — data layer поверх Supabase (events / artists / releases / streams / merch / users / auth / storage)
  - `script.js` — UI-логика публичного сайта
  - `admin/admin.js` — логика админ-панели
- `styles.css`, `admin.css` — стили публичной части и админки
- `vite.config.mjs` — конфиг Vite (multi-entry: `index.html` и `admin.html`)

### Как запустить проект локально (по-русски)

1. **Скачать репозиторий**
   - Через git:
     ```bash
     git clone <URL_ТВОЕЙ_РЕПЫ>
     cd npomelody-main
     ```
   - Или скачать ZIP с GitHub и распаковать в папку, затем в терминале перейти в неё:
     ```bash
     cd /путь/к/npomelody-main
     ```

2. **Установить зависимости**
   ```bash
   npm install
   ```

3. **Запустить dev-сервер**
   ```bash
   npm run dev
   ```

4. **Открыть сайт в браузере**
   - Публичный сайт: `http://localhost:5173/`
   - Админка: `http://localhost:5173/admin.html`

Если порт уже занят, Vite может предложить другой (например `http://localhost:5174`) — тогда использовать его.

### Продакшен-сборка

```bash
npm run build
```

Артефакты появятся в `dist/`. Их можно деплоить:

- на GitHub Pages (папка `dist` как содержимое ветки `gh-pages`);
- на любой статический хостинг (Netlify, Vercel Static, S3 и т.п.).

Базовый путь в Vite настроен как относительный (`base: './'`), поэтому сборка дружелюбна к GitHub Pages и подкаталогам.

### Как выложить сайт на GitHub Pages

1. **Собрать проект**
   ```bash
   npm run build
   ```
   После этого в папке `dist/` появится готовый к деплою сайт.

2. **Создать ветку `gh-pages` из содержимого `dist`**
   Выполнить из корня проекта:
   ```bash
   git add .
   git commit -m "build: обновить dist"   # если есть изменения

   git subtree push --prefix dist origin gh-pages
   ```
   Альтернативный вариант (через отдельную ветку вручную):
   ```bash
   git switch --orphan gh-pages
   git reset --hard
   cp -R dist/* .
   git add .
   git commit -m "deploy: первая версия сайта"
   git push -u origin gh-pages
   ```

3. **Включить GitHub Pages**
   - Открыть репозиторий на GitHub.
   - Зайти в **Settings → Pages**.
   - В разделе **Build and deployment**:
     - Source: **Deploy from a branch**
     - Branch: **gh-pages**, папка `/ (root)`.
   - Сохранить настройки и подождать 1–2 минуты, пока GitHub соберёт сайт.

4. **Открыть сайт**
   - Адрес будет вида: `https://<твой-логин>.github.io/<имя-репозитория>/`
   - Публичный сайт — корень.
   - Админка — `https://<твой-логин>.github.io/<имя-репозитория>/admin.html`

При каждом обновлении сайта:
- делаешь `npm run build`
- снова выполняешь `git subtree push --prefix dist origin gh-pages` (или свой выбранный сценарий),
- GitHub Pages автоматически обновит продакшн-версию.

### Supabase

Клиент и ключ Supabase сейчас захардкожены в `src/db.js` и работают через глобальный `window.supabase`:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`

На следующем этапе можно:

- вынести эти значения в окружение (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`) и прокидывать их через CI;
- дополнительно ужесточить RLS-правила в Supabase для таблиц.

### Сущности

Фронтенд и админка уже работают с основными сущностями:

- `events` — афиша / события (в админке есть CRUD, загрузка постеров в Supabase Storage)
- `artists` — артисты (CRUD + загрузка фото, флаг доступности для букинга)
- `releases` — релизы (рид-режим; редактор можно доработать)
- `streams` — трансляции / архив (рид-режим)
- `merch` — мерч (рид-режим)
- `users` — пользователи/участники (роль `admin/member`, управление ролями в админке)

Публичный сайт использует `window.dbLayer` (из `src/db.js`) для загрузки данных и авторизации участников (профиль + gated контент).

### Если браузер показывает «Опасный сайт»

Такое предупреждение (Google Safe Browsing) иногда появляется у легитимных сайтов с формами входа/регистрации — это ложное срабатывание. Что сделано в проекте:

- В `index.html` и `admin.html` добавлены мета-теги безопасности: **Content-Security-Policy** и **Referrer-Policy**. Они ограничивают источники скриптов и данных только своим сайтом и Supabase, что снижает риски и улучшает доверие со стороны браузера.

Если предупреждение всё равно показывается:

1. **Проверь домен** — возможно, помечен сам домен (например, GitHub Pages). Зайди в [Google Safe Browsing — сообщить об ошибке](https://safebrowsing.google.com/safebrowsing/report_error/) и укажи URL сайта, если уверен, что это ваш официальный сайт.
2. **Проверь Supabase** — если в консоли браузера видно блокировки запросов к `supabase.co`, проверь проект в панели Supabase (лимиты, подозрительная активность).
3. После деплоя подожди 1–2 дня — обновление списков Safe Browsing может занять время.
